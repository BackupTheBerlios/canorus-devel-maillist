Hi Matevz & Co,

I discussed with Georg about CAVoice/CAStaff dependencies and we thought
to implement a minimal control class with reference lists to f.e. clefs.

If the list gets changed by one class the other is notified about the
change. It might be useful (similar to transactions) to do several
changes at once.

A possible delay for the changes could be introduced if Canorus is in 
some kind of busy state (it would be more complex, so I'd introduce such 
delays only if there are really problematic performance issues).

Matevž Jekovec wrote:
> I think in the future we should move the shared signs outside the voice
> (to CAStaff directly). And I wouldn't use CAVoiceCtl and CAStaffCtl
> classes but put all the control stuff into the CAVoice and CAStaff
> itself (not many public control functions will exist I think - maybe 5,
> others are more model for insertion/deletion). This way we would
> simplify the score operations to the core classes CAVoice, CAStaff etc.
> No special control classes would be needed for the Python developer.
>
> I would create class CAQuantization for the tuplet and other quantiazion
> operations. I would create CAMainWinMode, CAMainWinInsert,
> CAMainWinDelete, CAMainWinEdit for the GUI operations specific to the
> current mode.

Insert should have two variants: Insert by moving like we already have
and also Insert by Overwriting like it is done already for x-tuples.
This would make it also easier for x-tuples requiring currently a very
exact mouse control to "overwrite" the pause elements by notes.

I don't exactly see the need for a Delete mode. You can delete both
in Insert and Edit mode (which replaces notes by pauses normally).
