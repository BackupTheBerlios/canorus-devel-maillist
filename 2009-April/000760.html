In theory, you could drive the interface via a Python script. As long as the classes behind it were C++, it would probably be reasonably fast. But I suspect that it would be better if Canorus had a consistent and solid UI, instead of a bunch of flexible, but incomplete scripts.

> Thanks to the plugin interface we mainly need to discuss how much to put

I suspect that it would be best to standardize on a single plugin language for "official" development. You end up getting into a sort of "Tower of Babel" prediciment. Even Microsoft, which pushed .NET on the idea that you could run various languages with it (IronPython, ASP.NET, etc.) has pretty much standardized around C# as the "one true language".

> Also I'm considering to go to a virtual machine like parrot if
> interfacing and/or performance improvements can be achieved this way
> (i.e. we only provide the interface and don't have to care about
> the language you want to implement your Canorus plugin).

Since it's bound to the same Canorus API underneath, and most scripting languages are pretty much identical, my preference as a developer is to use a language that has good intellisense support. In that regard, Python does poorly, because it doesn't require types to be declared. Without type declarations, the editor doesn't know what class to associate a variable to. You could look at the assignments and make a guess, I suppose.

Prolog is the exception to the rule, as far as scripting languages go. It's great for solving constraint-based problems (like voice leading, or even composing), but I wouldn't want to write UI code with it.
