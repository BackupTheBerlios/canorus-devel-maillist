Hi Matevž,

> Hi guys.
>
> As you know, our "context" describes the unlimited long structure which
> contains zero or many music elements. The term "context" was stolen from
> Lilypond and NoteEdit export to Lilypond. Looking further, "context" in
> lily does not necessarily describe this, but can be anything you want
> actually, so it's not the right word at all. My idea was to rename all
> "context" words to "track". This brings the following work:

Yes, I'm still pro renaming, as the word context in itself is very
abstract and I didn't hear about it before NoteEdit score editor (and I
have to pay attention because even here we often use the word context
in different semantic meanings)

> 1) CAContext ->  CATrack and other class renamings
> 2) getter/setter/query methods which contain word "context" ->  "track"
> 3) renamings of the files themselves

This far I see no problems. Could be easily done with sed.

> 4) renamings in Swig interfaces

The problem is here that existing plugins may break. We should therefore
deprecate this api for about 2-3 version before we remove it.

> 5) renamings in CanorusML file format - brings backward incompatibility.

Similar to above but not completely. Changes to the file formats might
happpen in the future too so it might be useful to provide functions for
doing the work in a way similar to lilypond but more powerful. So there
is f.e. a method convert70, maybe later convert80 which is called then
by convert70 when reading an old file. Of course the older the file the
slower the loading will be. But as it is only a one time thing so I
have no problems with this loading time.


> 5.1) create CanorusMLConverter class (which we eventually will need
> anyway) to convert from older versions of our format to the new one

see above.

> The job is pretty extensive, so I would like your opinions about this.
> Please write any other critics about the current API and score
> structure. Until we hit 1.0, we still have time to drastically change
> Canorus internals.

I only partially agree. I think after the next UI improvements
especially in the usability department there will be many more people
starting to use Canorus. So we should avoid drastic changes after that
version which might be released already in autumn (depending on my own
time and the time other developers can invest).

> The complete new score structure I had in mind can be found here:
> http://jekovec.net/tmp/scorediagram.png

Did you change more since LAC in Parma to the diagram ?
I haven't noticed any problems with it. Only thing is about the
controls.. there will be many more for inputing all kings of musical
elements. But I think it's better to model this in the context (not the
above one :) of each element itself, so it's much more MVC like.
