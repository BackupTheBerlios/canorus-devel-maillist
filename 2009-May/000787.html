> I was in contact with an developer via kguitar on sourceforge (see 
> https://sourceforge.net/tracker/index.php?func=detail&aid=2689386&group_id=7693&atid=357693 
> ).
> Parts of the kguitar that might not be useful to most Canorus user
> and should be provided in a plugin.
>
> Concerning MuseScore: We talked to Werner at the LAC in Parma and he
> told us that these things can be complicated because of the MuseScore
> code structure. I guess it's easy to make the logical, core part and
> maybe some control parts portable between applications.
>
> The user interface part though with layout/drawable also requires quite
> some work and here I currently don't see how MuseScore could profit from
> our layout view. It was a design decision by the MuseScore developers to
> combine (s)core and layout.
>
> We have longer plans to offer several libs that have parts of the
> Canorus code. The recent changes in the directory hierarchy work exactly
> in that direction. Mainly missing control classes (most controls are
> bound to CAMainWin currently) prevent to create them all.
>
> Borrowing Code from KGuitar would be mainly on the export/import layer
> as there is already quite some support for other tablature formats.
>
> Conclusion: Make the core/logic and other UI free code independent and
> useable in several projects. The UI part can be borrowed by Canorus or
> implemented by each program alone.
>
> Regards,
>
