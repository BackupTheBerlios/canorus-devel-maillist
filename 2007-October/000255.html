> It would also be nice to have better control of tempi, such as long
> accelerandos or decelerandos that are not just linear.  For example, an
> accelerando that starts gradually and speeds up much faster at the end,
> or a change in tempo that speeds up and slows down in waves over the
> course of the whole piece of music.  Such an implementation might also
> allow for different styles of playing, such as the swing style one
> person was looking for.
>   
I'm not sure if this is a good thing for Canorus itself. My though
about it that Canorus itself is mainly a score editor. It has some
midi capabilities for record and for playback (to control thing you
have composed). OTOH I don't know that many contemporary
music pieces having such features. I'd beg to differ between
interpretation and composition. My idea to this would be that if it
is not completely against the CAPlayable model it should be done as
an external plugin. I'd even vote for a replacement for certain models
either as plugins or as modules if you wish to do some special thing
normal users usually don't do.
So my idea is that when you load such a plugin Canorus somehow replaces
the default playable model by your own. Of course to realize that you need
to define an abstract interface for both models (if not already done).

I could imagine similar interfaces for other parts of the Canorus model
as well.
The main issue I see here is stability of Canorus as a whole. Maybe to
achieve
this we need to be forced to create base classes implementing the common
features and the default model implements only the default behaviour.

What do you think about this idea ?
