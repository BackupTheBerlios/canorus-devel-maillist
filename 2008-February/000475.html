Hi,

So I've been considering whether use threads or processes, and how to do 
it. It seems that calling scripts as new process using swig is too 
complicated. Let's use threads. If something happens with pycli or other 
plugin, thread containing it should be killed [and created new, 
depends]. I'm not going to use signals either.

Idea1: I want for canorus to provide plugin depencies, for example: 
plugin A wants to get input using an inputbox, and plugin B provides 
this functionality...so plugin A tells canours to call plugin B, to get 
that input (I think making this in PyQT would be easier, than in c++/qt; 
so that's why is it in plugin).

Another idea: there's a site mutopiaproject.org, there are many 
classical GPLd sheets. So let's make a plugin, that would download 
sheets from there automatically, convert from lilypond to canorus 
format. (a depency) and open that sheet => to make it as easy [and 
simple!] as possible for the user.

I think the CLI could help very much. Let's say you want to call 
function in a plugin, so you write in the cli: "/callfunction 'plugin' 
'function' arguments, and you can debug the functions. Or you can edit 
the plugin without closing canorus...(/recompile 'plugin' .py -> pyc; 
don't know much about ruby), I hope it's possible. (internal CLI 
commands would start with "/").

That are some of my ideas. If you have any other, post them.

Regards,
Stefan
