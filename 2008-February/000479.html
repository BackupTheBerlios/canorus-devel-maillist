> I was learning/examining the possibilities for plugin architecture to
> work as good as possible. At first I thought executing plugins in a
> process would be safer but now I guess it's unnecessarily complicated.
> So threads are ok (or not?...).

For me I'd prefer processes all the except being extremely careful about
synchronisation stuff.

> When calling a python function from c, the python module is reloaded
> every time a function is called => so it does mean, that the
> particular python module would not retain it's internal state. So if
> the module from ImportModule will be preserved, would it retain it's
> global variables and stuff?

If you create a python class and make the object from that class global
the content from that object is being kept between reloads.

Well I didn't yet put too much thought into it, but I hope that the workshop
or some discussion here will point in the direction (or maybe several ones
if we need them for flexibility) to go for communication and running plugins.

As concept I guess we have following questions to answer (feel free to add
your own questions concerning plugins <-> canorus)

- How to communicate between plugin and Canorus for simple notifying
  things (observer -> signal/slot)
- How to communicate more complex objects (based on widgets) from canorus
  to the plugin
- How to integrate more complex widgets from plugins into canorus
- How to let plugins run in parallel to the canorus process
