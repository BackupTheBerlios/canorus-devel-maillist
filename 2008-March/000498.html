Maybe it's theoratically possible to have several interpreter objects in
one thread but they either inhibit it or just say it doesn't work because
these objects aren't thread-safe. I guess it would be a separate challenge
to add thread safety to such an object ;) (Except the general challenge due
to the complexity of the interpreter I'd estimate some performance hits).

On the Canorus part my wish is in a first step to have some simple
communication possibilities in both directions, I'd prefer signal-slots
of course.

The other more complicated wish is to be able to integrate python views
into a Canorus widget (or some way to ease the creation of the view content
for a python script). It's be also nice to have it the other way round though
I'd need to think of a feature where this would be desirable.

For simple UI stuff (forms, dialogs) with existing Qt widgets I will create
a generic class having methods to create such a form (including layout). For
this class being useful to scripts we'd need the above communication layer
with the python script.

One thing being more a sort of requirement would be that all this is not too
much tight to python itself, so we can adapt it to ruby later too.

As there are only 3 months of time including documentation and all the
management work I'd first concentrate on your own ideas before starting to
work on my wishes. It would be good though to keep the wishes in your mind
so we can later expand on your work.

I'll set up a Canorus SoC page too for the PSF soon for the PSF developer
to get easier into the Canorus project themselves.
